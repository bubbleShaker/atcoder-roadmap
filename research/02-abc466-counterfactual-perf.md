# ABC466：「C / D を通していたらパフォいくつだったか」を順位表復元で実測する

関連: #18 ／ 本番の記録: [logs/2026-W28.md](../logs/2026-W28.md) ／ 再現: `python tools/counterfactual_perf.py abc466 --start "2026-07-11 21:00" --minutes 100 --user Coji --scenario 600:30`

## 結論（先に）

**C を通しても、パフォは 634 が天井で、レートは下がる。**
レートが上がるのは **D まで通したとき（perf 900〜1050）だけ**。

つまり ABC466 で失ったのは「C の 300点」ではなく、**C に溶かした時間で失った D**。

## ここから読めること

### 1. 3完のパフォは「速さ」でほとんど動かない

C を **20分**で通しても **59分**で通しても、パフォは **632 → 606** で **26 しか変わらない**。
極端に、開始 **1分**で 3完しても **634** にしかならない（+2）。
3完の人が 6500〜6700位に密集しているため、この帯では時間短縮のリターンがほぼ無い。

**「C を速く抜ける」ことの価値は、パフォそのものではなく、D に使える時間として現れる。**
PLAN.md の M1（速度と一発AC）は**目的ではなく手段**である、ということが数値で裏付けられた。

### 2. D を通すかどうかが崖

600点 → 1000点で、順位は **6535位 → 4058位（約2500人ぶん）**、パフォは **628 → 1017（+389）**。
**D の1問だけで、C の1問の約2.5倍のパフォが動く。**

しかも **終了間際（99分）に滑り込んでも perf 896** ある。
D は「時間内に間に合わせさえすれば」十分すぎるリターンがある。

### 3. 今回の「取れた最高」は perf 約1000

本番の事実として、

- **C は解法が合っていた**（質問回数 2N の見落としで WA。→ [knowledge/02](../knowledge/02-interactive-query-limit-abc466c.md)）
- **D も方針が合っていた**（削除処理の実装だけが壊れていた。→ [knowledge/03](../knowledge/03-placing-rooks-abc466d.md)）

つまり **両方とも「解けるはずだった問題」**。C を 30分で抜けて D を 65〜75分で通す展開は現実的な射程内で、
そのときのパフォが **約 980〜1020**。これが今回の**実質的な天井**。

**実際の 463 との差は、およそ 550**。

### 4. レートが上がる条件

現レート 854 に対し、レートが上がるにはパフォがそれを上回る必要がある。
**3完（開始1分で解いても 634）ではレートは上がらない。** D を通して初めて 900 を超える。

> PLAN.md の M2「D を毎回通す」が、レート改善の**唯一の直接的なレバー**であることが確認できた。
> M1（Cを速く）と M4（爆死を消す）は、そのための時間を作る手段として効いている。

## 方法（なぜこの数字を信用してよいか）

### データ源

| 欲しいもの | 入手先 | 可否 |
|---|---|---|
| 順位表（誰が何点を何分で） | `atcoder.jp/contests/abc466/standings/json` | **ログイン必須で取得不可**（302） |
| 順位 → パフォの対応 | `atcoder.jp/contests/abc466/results/json` | 公開。全13301人ぶん取得 |
| 全提出（時刻・結果） | `kenkoooo.com/atcoder/atcoder-api/v3/from/<epoch>` | 公開。コンテスト中の **68,639件** |

順位表そのものが取れないので、**提出ストリームから順位表を復元**した。

### 復元の規則（AtCoder の順位規則）

- 得点 = AC した問題の配点合計（実測: **A100 / B200 / C300 / D400 / E450 / F500 / G600**。E・F・G は「よくある配点」と違うので問題ページから取得した）
- 時間 = 最後の AC までの経過 + **300秒 × (AC した問題での誤答数)**（未 AC の問題の誤答はペナルティに数えない／CE は数えない）
- 順位 = 得点降順 → 時間昇順、同着は同順位

### 補正：提出データが欠けている参加者が20人いる

AtCoder Problems 側に**提出が1件も存在しない参加者が20人**いた（AtCoder 側には順位もパフォも付いている実在の参加者。ユーザー別 API でも 0 件）。
逆に、提出はあるが公式結果に居ない人が 2人。

この20人を無視すると「自分より上にいる欠損者の数」だけ順位が良く出る
（実際、下位ほど誤差が増え、**欠損者の人数である +20 で頭打ち**になる形が観測された）。
ただし**欠損者の公式順位は results/json から分かる**ので、復元側の順位に「自分より上にいる欠損者の数」を足し戻せば公式順位を再現できる。

**欠損者の得点・時間は分からないため、「自分と同着なのか1つ上なのか」だけは原理的に判定できない。**
そこで順位は範囲で計算している（重なるときだけ幅1が付く。それ以外は一意）。

### 検証

**全 11,077人**について、公式順位が復元した範囲に収まることを確認した（うち幅1になったのは 23人のみ、残りは一意に的中）。
Coji の実績も **300点 / 8分04秒 → 7634位 / perf 463** と**公式に完全一致**。

> 検証が通らなければスクリプトは結果を出さずに中止する（`build_verified_standings`）。
> 上の数字は、この検証を通過したうえで出ている。

この検証は順位規則を実際に識別している（reviewer が反証実験で確認）:
CE をペナルティ扱いに変えると **10,162人**が、未 AC 問題の誤答をペナルティに数えると **10,660人**がミスマッチして中止する。

順位規則の実装と、欠損者補正の導出は `tools/test_counterfactual_perf.py`（14件、ネットワーク不要）で固定してある。

## 制約・注意

- 反実仮想は「**その得点・その時刻で提出したときの、実際の順位表における位置**」であり、他の参加者の行動は変わらない前提。
  現実には Coji が D を通していれば他の人の順位も1つずつずれるが、影響は1人ぶんなので無視できる。
- 「600点 / 30分」の 30分は**ペナルティ込みの時間**。WA を2回出して 30分で AC した場合は `600:40` で評価すること。
- パフォは results/json の (順位, パフォ) を**実測値として線形補間**している（推定式は使っていない）。
- レート変化は算出していない（AtCoder のレートは全参加履歴の加重平均のため、1回ぶんの寄与を単独で決められない）。
  ここでは「パフォが現レート 854 を上回るか」だけを判断材料にしている。

# 「直前の1個」ではなく「対応関係」を持つ（ABC466 D）

対象: [ABC466 D - Placing Rooks](https://atcoder.jp/contests/abc466/tasks/abc466_d)
関連: #16 ／ 本番の記録: [logs/2026-W28.md](../logs/2026-W28.md)

## 問題

N×N のグリッド。M 回の操作を順に行う。操作 i は次の 3 手順:

1. 行 R_i 上の駒をすべて取り除く
2. 列 C_i 上の駒をすべて取り除く
3. マス (R_i, C_i) に駒を置く

M 回終わった後に盤上に残っている駒の数を答える。N, M ≤ 3×10⁵。

## 本番でやったこと（未提出・サンプル2 で WA）

「盤面は常に **1 行に高々 1 個・1 列に高々 1 個**しか駒がない（各操作が行と列を先に空にするから）
→ したがって **駒数 = 占有している行数**」という方針を立て、`row[]` / `col[]` の bool 配列と、
**直前の操作で置いた駒**の位置 `b_row` / `b_col` を持って更新しようとした。

## 方針は正しい。壊れていたのは削除処理

**「駒数 = 占有行数」という着眼は正しい**（各操作が行 R_i を空にしてから 1 個置くので、どの行にも高々 1 個しか存在しえない）。

問題は `b_row` / `b_col` が **直前の操作で置いた駒 1 個しか覚えていない**こと。
実際には、行 R_i や列 C_i には**もっと前の操作で置かれた駒**が居る可能性がある。それを消し損ねる。

サンプル2 `N=2, (1,2) → (2,1) → (1,1)` で顕在化する:

| 操作 | 盤面 |
|---|---|
| (1,2) | {(1,2)} |
| (2,1) | {(1,2), (2,1)} |
| (1,1) | 行1 を消す → **(1,2) が消える**／列1 を消す → (2,1) が消える／(1,1) を置く → **{(1,1)}** |

答えは 1。ところが本番のコードは `b_row=2, b_col=1`（＝2手目の駒）しか見ていないので、
**1手目の (1,2) を消し損ね**、`row[1]` と `row[2]` が両方 true のまま残って **2 を出力**した。

## 直し方A（本番の方針を活かす最小修正）

「直前の 1 個」ではなく、**行と列の対応関係そのもの**を持つ。

- `colOf[r]` = 行 r にある駒の列（無ければ空）
- `rowOf[c]` = 列 c にある駒の行（無ければ空）

操作 (r, c) では:

1. 行 r に駒があるなら（`colOf[r]` が空でないなら）、その駒を消す。
   このとき **`colOf[r]` と `rowOf[colOf[r]]` の両方**をクリアする（行側だけ消すと、列側に幽霊が残る）
2. 列 c に駒があるなら（`rowOf[c]` が空でないなら）、同様に **`rowOf[c]` と `colOf[rowOf[c]]` の両方**をクリアする
3. `colOf[r] = c`、`rowOf[c] = r` と結び直す

最後に**占有している行の数**を数えれば答え。O(N + M)（カウンタを持てば O(M)）。

> **実装の落とし穴**: 手順1と手順2は独立ではない。既に (r, c) に駒がある場合、手順1が `rowOf[c]` を消してしまうので、
> 手順2はもう「列 c に駒は無い」と判断する。結果は正しくなるが、**手順1で読んだ値を手順2でも使い回そうとすると壊れる**。
> 各手順で必ず**その時点の**配列を読み直すこと。

> 覚え方: 「行を消す」には、その行の駒が **どの列を握っていたか** を知らないといけない。だから対応表が要る。

## 直し方B（言い換えると、もっと簡単になる）

操作 i で置いた駒が**最後まで生き残る条件**を直接考える。

後続の操作 j (> i) は「行 R_j」と「列 C_j」を空にする。したがって駒 (R_i, C_i) が消されるのは、
**R_j = R_i または C_j = C_i となる j > i が存在するとき**、かつそのときに限る。

つまり生き残る条件は:

> **i が「行 R_i の最終出現」であり、かつ「列 C_i の最終出現」でもある**

`lastRow[r]` / `lastCol[c]`（その行・列が最後に登場した操作番号）を先に求めておき、
`lastRow[R_i] == i && lastCol[C_i] == i` を満たす i を数えるだけ。O(N + M)、状態更新なし。

> **同じマスに置き直された場合**（j > i で R_j = R_i かつ C_j = C_i）も破綻しない。
> i は行 R_i の最終出現ではなくなるので数えられず、代わりに j 側が 1 個として数えられる。**二重計上も数え落としも起きない。**
> また、生き残る駒どうしは行も列も相異なるので（各行・各列に高々1個）、重複して数えることもない。

## 気づき

- 本番の方針（占有行数を数える）は**結論として正しかった**。詰め切れなかったのは**状態の持ち方**であって、考察ではない。
- 「直前の 1 個だけ覚える」で済むのは、**過去の駒が高々 1 個しか影響しない**ときだけ。今回は行と列の**2 方向**から過去の駒が効いてくるので、対応表が要る。
- **削除操作は「その要素を消すのに必要な情報が手元にあるか」を先に確認する**。無ければ、その情報を持つデータ構造を先に決める。
- 逆順・最終出現で考えると状態管理が消えることがある（直し方B）。**「時系列で状態を更新する」で詰まったら「後ろから見て生き残る条件」に言い換えてみる**のは汎用的に効く。

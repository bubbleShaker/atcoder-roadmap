# 貪欲が壊れる形：修正が「隣」に波及するなら、自由度は先頭だけ（ABC467 C）

対象: [ABC467 C - Adjacent Sums (easy)](https://atcoder.jp/contests/abc467/tasks/abc467_c)（300点）
関連: [#23](https://github.com/bubbleShaker/atcoder-roadmap/issues/23) ／ 本番の記録: [logs/2026-W29.md](../logs/2026-W29.md)

## 問題

長さ N の数列 A（2 ≤ N ≤ 2×10⁵）と長さ **N-1** の数列 B。
操作は「i を選び **A_i に 1 を加える**」を任意回数（加算時に mod は取らない）。
すべての i で $(A_i + A_{i+1}) \bmod M = B_i$ にするための **最小操作回数**を求める。
**M = 2 固定**（E が [Adjacent Sums (hard)](https://atcoder.jp/contests/abc467/tasks/abc467_e) で一般の M。C が M=2 固定なので E は一般 M を扱う側と推測されるが、E の制約は未確認）。

## 本番でやったこと（WA・未AC）

**前から貪欲に直した。** 左から i を見て、$(A_i + A_{i+1}) \bmod M \ne B_i$ なら
$A_i$ か $A_{i+1}$ の **その場で得な方**に +1 する。

その場では必ず条件を満たせるので、一見正しく見える。

## なぜ落ちたか — 直した先が「前」を壊す

$A_i$ は **2つのペア**（$i-1$ 番目と $i$ 番目）に同時に効く。
だから「$i$ 番目のペアを直すために $A_i$ を +1 する」と、**すでに確定させたはずの $i-1$ 番目のペアが壊れる**。
左から進む貪欲は振り返らないので、**壊したことに気づかないまま操作回数を数え終える**。

### 最小の反例

`A = (0, 0, 0, 0)` / `B = (0, 1, 0)` → **貪欲は 1、正解は 2**

| i | 状態 | 判定 |
|---|---|---|
| 1 | $(A_1+A_2) = 0 = B_1$ | OK |
| 2 | $(A_2+A_3) = 0 \ne B_2 = 1$ → **$A_2$ に +1**（コスト1）。A=(0,**1**,0,0) | ここで $(A_1+A_2) = 1 \ne B_1 = 0$ となり **1番目が壊れる**が、もう振り返らない |
| 3 | $(A_3+A_4) = 0 = B_3$ | OK |
| | **出力 1**（実際には条件を満たしていない） | 正解は **2** |

## 全数チェックの実測 — 「たまに落ちる」ではない

貪欲の「どちらに +1 するか」の決め方を3通り実装し、N=2〜7 の入力を全数照合した（下記コード）。

| 版 | N=3 | N=4 | N=5 | N≤7 累積 | 誤差の向き |
|---|---|---|---|---|---|
| **その場で得な方**（本番と同種） | 0 / 32 | **16 / 128** | 64 / 512 | **2768** | **全件が過小** |
| 常に左（$A_i$） | 8 / 32 | 64 / 128 | 320 / 512 | 7368 | 過小1552 / 過大5816 |
| 常に右（$A_{i+1}$） | 8 / 32 | 16 / 128 | 160 / 512 | 3384 | **全件が過大** |

本番の実装は残っていないが、**「次のペアも満たせる側を優先し、無理なら左」で再現したところ WA 数が一致**した
（本人の報告と同じ **2768**、最初の反例も同じ `A=(0,0,0,0) B=(0,1,0)`）。
N=4 だけで **128通り中16通り**が落ちる。**構造的に間違っている**解法だった。

> この版では N=3 に反例が無いので、上の入力は**正真正銘の最小反例**。
> ただし「常に左」「常に右」に固定した版は N=3 から落ちる（`A=(0,0,0) B=(1,1)` など）。
> **どう選んでも壊れる** = 選び方の問題ではなく、貪欲という枠組みの問題。

### 誤差の向きが版によって違うのが本質

- **前を壊す版（得な方・常に左）は「過小」に出る** — 壊れたペアを直したまま数えないから
- **前を壊さない版（常に右）は「過大」に出る** — 条件は必ず満たすが、最小にならないから

そして「常に右」が最小にならない理由が、そのまま正解への入口になる。
**右にしか加えないと $A_1$ は初期値のまま動かない。** つまりこの版は
**「$A_1$ を今の値に固定した世界」を1つ計算しているだけ**で、
本来ありうる **もう1つの世界（$A_1$ の偶奇を変えた場合）を見ていない**。

> **貪欲を正しく（前を壊さないように）やると、それは $A_1$ 決め打ちの1通りと同じものになる。**
> 足りないのは「**もう1通りも試して min を取る**」ことだけだった。

## 正しい発想 — 自由度は $A_1$ しかない

M=2 なので、余りは「**偶数か奇数か**」だけ。条件は次の意味になる。

- $B_i = 0$ → $A_i$ と $A_{i+1}$ は **同じ偶奇**
- $B_i = 1$ → $A_i$ と $A_{i+1}$ は **違う偶奇**

つまり **隣り合う要素の偶奇の関係が全部指定されている**。
$A_1$ の偶奇を決めた瞬間、$A_2$ が決まり、$A_3$ が決まり……と **最後まで一意に確定する**（$A_{i+1} \equiv B_i - A_i \pmod 2$）。

**選択の余地がどこにも無い。** だから貪欲（＝その場で選ぶ）という枠組み自体が問題に合っていない。

**あり得る世界は $A_1$ が偶数か奇数かの2通りだけ。** 各世界で、
目標の偶奇と現在の値が違う要素にだけ +1 を1回（$0\to1$ も $1\to0$ もコスト1）。
その合計を2通りで比べて **min** を取る。計算量 O(N)（一般の M なら決め打ちが M 通りで O(NM)）。

反例で確認すると:

| $A_1$ の決め打ち | 目標の偶奇 | コスト |
|---|---|---|
| 偶（0） | (0, 0, 1, 1) | 0+0+1+1 = **2** |
| 奇（1） | (1, 1, 0, 0) | 1+1+0+0 = **2** |

→ 答え **2**。貪欲の 1 は不足していた。

## この形の見分け方（次に同じ構造を見たとき）

**「1つの要素が複数の制約に共有されている」なら、局所的な修正は他を壊す。**

- $A_i$ が $i-1$ 番目と $i$ 番目のペアに**両方**現れる → 片方を直すともう片方が動く
- こういう連鎖では、**自由に決められるのは端（先頭）だけ**で、残りは従属する
- したがって「**先頭の値で場合分けして全探索 → 残りは決定的に埋める**」が定石

> **チェック**: 「その場の最善」を選ぼうとしたとき、**選んだ結果が既に確定させた場所に影響するか**を必ず自問する。
> 影響するなら貪欲は使えない。**探索空間を先頭の自由度に畳めないか**を考える。

「全探索の感覚」を思い出すのが正解筋だった、という回。

## 反例の炙り出し方（今回いちばん再現性がある道具）

**N を小さくして全パターンを回し、自分の解と正解（愚直解）を diff する。**
N=4・値 0/1 なら $2^4 \times 2^3 = 128$ 通りしかない。**速度は不要、正しさだけあればよい。**

```python
from itertools import product

def correct(A, B):
    """A_1 の偶奇を決め打つと残りは一意。2通りの min。"""
    best = None
    for first in (0, 1):
        want = [first]
        for b in B:
            want.append((b - want[-1]) % 2)
        cost = sum(1 for a, w in zip(A, want) if a % 2 != w)
        best = cost if best is None else min(best, cost)
    return best

def greedy(A, B):
    """左から見て、合わないペアが出たら「次も満たせる側」に +1（無理なら左）。"""
    a = list(A)
    cost = 0
    for i in range(len(B)):
        if (a[i] + a[i + 1]) % 2 == B[i]:
            continue
        if i + 2 <= len(B) and (a[i + 1] + 1 + a[i + 2]) % 2 == B[i + 1]:
            a[i + 1] += 1
        else:
            a[i] += 1
        cost += 1
    return cost

for n in range(2, 8):
    for A in product((0, 1), repeat=n):
        for B in product((0, 1), repeat=n - 1):
            if greedy(A, B) != correct(A, B):
                print(n, A, B, greedy(A, B), correct(A, B))
                break
```

**WA の原因は頭で探すより機械で探す方が速い。** とくに「解法は合っているはず」と思い込んでいるときに効く。
今回はこれで最小反例が出て、そこから手で追える形になった。
